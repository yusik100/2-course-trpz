## 2. Документація по розробленому веб-застосунку

### 2.1. Призначення застосунку
**Simple Inventory** (`mywebapp`) — це мікросервіс для ведення простого обліку обладнання. Застосунок дозволяє переглядати список інвентарю, додавати нові предмети та отримувати детальну інформацію про конкретний предмет. Реалізовано Content Negotiation: залежно від заголовку `Accept`, API віддає дані у форматі `application/json` або `text/html`.

### 2.2. Налаштування середовища (для локальної розробки)
Для запуску застосунку без автоматизованого скрипта необхідні:
1. Встановлений Node.js (v18+) та npm.
2. Встановлений сервер MariaDB.
3. Створена база даних та користувач із правами доступу до неї.

### 2.3. Запуск веб-застосунку (вручну)
Перейдіть у директорію `mywebapp` та встановіть залежності:
```bash
npm install
```

Виконайте міграцію бази даних (створення таблиць):
```bash
node migrate.js --dbuser=<user> --dbpass=<password> --dbname=<database>
```

Запустіть застосунок:
```bash
node mywebapp.js --port=8000 --dbuser=<user> --dbpass=<password> --dbname=<database>
```

### 2.4. Документація по API-ендпоінтах
* **GET /** — Кореневий ендпоінт. Віддає text/html зі списком доступних маршрутів.
* **GET /items** — Вивести список усіх предметів в інвентарі (id, name). Підтримує JSON та HTML.
* **POST /items** — Створити новий запис (очікує параметри name та quantity). Підтримує JSON та HTML.
* **GET /items/:id** — Вивести детальну інформацію по запису (id, name, quantity, created_at). Підтримує JSON та HTML.
* **GET /health/alive** — Перевірка стану (повертає 200 OK).
* **GET /health/ready** — Перевірка підключення до БД (повертає 200 OK або 500 Error).

## 3. Документація по розгортанню

### 3.1. Базовий образ віртуальної машини
* Для розгортання використовується офіційний образ Debian 12.
* Рекомендується завантажити debian-12.x.x-amd64-netinst.iso з офіційного архіву образів Debian (версія трошки застаріла) https://cdimage.debian.org/cdimage/archive/
* Перейдіть за посиланням, знизу у списку папок знайдіть одну з версій 12-ї гілки (наприклад, 12.8.0 або 12.9.0).
* Зайдіть в цю папку, потім натисніть amd64 -> iso-cd.
* Там, у самому низу списку файлів, буде лежати debian-12.x.x-amd64-netinst.iso (наприклад ось https://cdimage.debian.org/cdimage/archive/12.9.0/amd64/iso-cd/)

### 3.2. Вимоги до ресурсів віртуальної машини
* **CPU:** Мінімум 2 ядра.
* **RAM:** 2048 MB (рекомендується 4096 MB для стабільної роботи GUI).
* **Disk:** Мінімум 10 GB.
* **Video:** 128 MB.

### 3.3. Встановлення ОС
Спеціальних налаштувань не потрібно. Встановлення проводиться за замовчуванням (з підключенням до мережі Інтернет для завантаження пакетів). Під час встановлення створюється користувач vboxuser (або інший), який згодом буде заблокований автоматизацією.

### 3.4. Вхід на ВМ та credentials
Вхід здійснюється через консоль VirtualBox (GUI).
* **До виконання автоматизації:** користувач root (або ваш дефолтний користувач) із вказаними вами при встановленні паролями.
* **Після виконання автоматизації:** вхід здійснюється за допомогою створених користувачів (дефолтний користувач блокується). Пароль для всіх нових користувачів за замовчуванням — 12345678 (потребує зміни при першому вході).

### 3.5. Як запустити автоматизацію розгортання
Увійдіть в систему під користувачем root (або через sudo) та виконайте:

```bash
1. Встановіть git:
apt update
apt install -y git

2. Склонуйте репозиторій:
git clone https://github.com/yusik100/2-course-trpz.git

3. Перейдіть до папки скрипта і запустіть його:
cd 2-course-trpz/lab1
bash setup.sh
```

Скрипт автоматично встановить Nginx, MariaDB, Node.js, створить користувачів, налаштує Systemd Socket Activation та Reverse Proxy.

## 4. Інструкція з тестування розгорнутої системи

Після успішного виконання скрипта setup.sh, коректність налаштувань можна перевірити наступними кроками:

**1. Перевірка роботи застосунку та Nginx (Reverse Proxy):**
`curl -H "Accept: application/json" http://localhost/items`
Очікуваний результат: Повернення пустого масиву [] або списку об'єктів у форматі JSON. Це підтверджує, що Nginx (порт 80) успішно проксіює запити на Socket (порт 8000), який пробуджує застосунок і підключається до БД.

**2. Перевірка безпеки Nginx:**
`curl -I http://localhost/health/alive`
Очікуваний результат: Помилка 403 Forbidden (доступ до health-ендпоінтів ззовні заблоковано).

**3. Перевірка Systemd Socket Activation:**
`sudo systemctl status mywebapp.socket`
Очікуваний результат: Статус active (listening) і вказівка Triggers: mywebapp.service.

**4. Перевірка користувачів та їх прав:**
* **su - student:** Має перевести на користувача, вимагати зміну пароля. Доступна команда sudo ls /root. У домашній директорії знаходиться файл gradebook із цифрою 8.
* **su - teacher:** Вимагає зміну пароля. Має права адміністратора (sudo). Файл gradebook відсутній.
* **su - operator:** Дозволено виконувати команду sudo systemctl reload nginx та керувати станом сервісу mywebapp без введення пароля. Спроба виконати sudo ls /root або sudo systemctl stop nginx блокується системою (Permission denied).
* **sudo su - app:** Миттєве закриття сесії без помилок доступу, оскільки користувач має оболонку /bin/false (захист від входу людини), але від його імені успішно працює фоновий процес застосунку (ps -u app -f).

# Лабораторна робота №2: Контейнеризація

## Запуск проєкту за допомогою Docker Compose

Цей проєкт повністю контейнеризовано. Автоматичний запуск та координація всіх пов'язаних сервісів (веб-застосунок, веб-сервер Nginx та база даних PostgreSQL) здійснюється за допомогою інструменту **Docker Compose** в окремій ізольованій мережі.

### Попередні вимоги

Перед запуском переконайтеся, що у вашій системі встановлено та запущено:
* **Docker**
* **Docker Compose** (включений за замовчуванням у Docker Desktop)

### Інструкція із запуску

1. Перейдіть у кореневу директорію проєкту (де розташований файл `docker-compose.yml`).

2. Виконайте команду для завантаження базових образів, збірки локальних Dockerfile та запуску контейнерів у фоновому режимі:
   `docker compose up -d --build`
   * *Прапорець `--build` примусово перезбирає локальні образи застосунків, гарантуючи актуальність коду.*
   * *Прапорець `-d` (detached mode) запускає контейнери у фоні, залишаючи термінал вільним.*

3. Перевірте статус та працездатність усіх піднятих сервісів:
   `docker compose ps`
   Усі три сервіси (`web`, `nginx`, `db`) повинні перебувати у статусі `Up`.

### Корисні команди для розробки

* **Перегляд логів** у реальному часі для всіх сервісів:
  `docker compose logs -f`
* **Перегляд логів для окремого сервісу** (наприклад, веб-додатка):
  `docker compose logs -f web`
* **Перезапуск конкретного сервісу** (корисно при оновленні коду без зупинки всього оточення):
  `docker compose restart web`

### Зупинка сервісів

Щоб зупинити роботу системи та очистити створені Docker-мережі, виконайте:
`docker compose down`

* Завдяки використанню іменованих томів (Docker Volumes), дані бази даних зберігаються на хост-машині локально. При виконанні команди `docker compose down` інформація в БД **не втрачається** і буде повністю доступна при наступному піднятті контейнерів за допомогою `docker compose up`.

## Відступ від початкових вимог методички

Під час виконання лабораторної роботи я зіткнулася з критичними проблемами при спробі розгорнути інфраструктуру локально (Windows + VirtualBox + Terraform). Неофіційні провайдери VirtualBox для Terraform мають погану сумісність із мережевим стеком Windows (проблеми з bridged adapters) та некоректно обробляють cloud-init для створення користувачів та прокидання SSH-ключів. 

Щоб виконати головну мету роботи — **засвоєння принципів декларативного керування інфраструктурою** — було прийнято інженерне рішення перенести розгортання в реальне хмарне середовище **Google Cloud Platform (GCP)**. Це дозволило повноцінно реалізувати IaC підхід, налаштувати безпечні правила Firewall та створити ізольовану VPC-мережу, що відповідає сучасним стандартам.

## Архітектура системи

Система складається з двох віртуальних машин (Ubuntu 22.04 LTS), розміщених в одній приватній мережі (VPC) Google Cloud.
```
          +--------------VM1 (worker)---------------+    +---VM2 (db)---+ 
client -> | nginx (0.0.0.0:80) -> web app (:8000)   | -> | MariaDB      | 
          +-----------------------------------------+    +--------------+
```

### Мережеві обмеження (Firewall & Security)
* Web App (Node.js): Слухає виключно 127.0.0.1:8000. Прямий доступ ззовні неможливий.
* База даних (MariaDB): Слухає внутрішній IP (10.186.0.3). Доступ до порту 3306 дозволений через GCP Firewall *тільки* для машин з тегом web-server.
* Nginx: Виступає як Reverse Proxy, приймає підключення на 0.0.0.0:80 і перенаправляє їх на локальний застосунок.

## Інструкція із запуску

### 1. Розгортання інфраструктури (Terraform)
Перейдіть до директорії terraform/. Перед запуском переконайтеся, що ви авторизовані в GCP CLI.

- Крок 1. У файлі main.tf замініть значення змінної project_id (YOUR_PROJECT_ID) на ID вашого проєкту GCP.
- Крок 2. Ініціалізуйте Terraform:
    terraform init
- Крок 3. Запустіть створення ресурсів:
    terraform apply
Після успішного виконання Terraform виведе 3 IP-адреси: worker_public_ip, db_public_ip та db_internal_ip. Збережіть їх.

### 2. Налаштування конфігурації (Ansible)
Перейдіть до директорії ansible/.

- Крок 1. Відкрийте файл inventory.ini та замініть заглушки на реальні IP-адреси, які видав Terraform.
- Крок 2. Запустіть плейбук:
    ansible-playbook -i inventory.ini playbook.yml
    
Примітка: Усі таски написані з дотриманням принципу ідемпотентності. Повторний запуск плейбука не призведе до жодних змін (changed=0).

## Критерії приймання та інструкція для перевірки

Нижче наведені команди для перевірки відповідності системи вимогам лабораторної роботи.

### 1. Health Checks та зв'язок між вузлами

### 2. Користувачі та їх права
Підключіться до Worker-вузла по SSH і перевірте створених користувачів:

Перевірка користувача app (Мінімальні права):

Очікуваний результат: в списку є процес node.

Перевірка користувача operator (Обмежений sudo):

    sudo -su operator

    # Команда вище видасть помилку: user not allowed

Перевірка користувача teacher (Sudo з паролем):

    sudo -su teacher
    sudo cat /etc/shadow
    # Система запитає пароль (введіть 12345678)

### 3. Мережева безпека та розподіленість

Очікуваний результат: Connection timed out (Firewall блокує доступ).

Веб-застосунок не стирчить назовні:

Очікуваний результат: Connection refused (Доступ тільки через 80 порт Nginx).
