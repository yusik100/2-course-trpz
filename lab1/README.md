## 2. Документація по розробленому веб-застосунку

### 2.1. Призначення застосунку
**Simple Inventory** (`mywebapp`) — це мікросервіс для ведення простого обліку обладнання. Застосунок дозволяє переглядати список інвентарю, додавати нові предмети та отримувати детальну інформацію про конкретний предмет. Реалізовано Content Negotiation: залежно від заголовку `Accept`, API віддає дані у форматі `application/json` або `text/html`.

### 2.2. Налаштування середовища (для локальної розробки)
Для запуску застосунку без автоматизованого скрипта необхідні:
1. Встановлений Node.js (v18+) та npm.
2. Встановлений сервер MariaDB.
3. Створена база даних та користувач із правами доступу до неї.

### 2.3. Запуск веб-застосунку (вручну)
Перейдіть у директорію `mywebapp` та встановіть залежності:
```bash
npm install
```

Виконайте міграцію бази даних (створення таблиць):
```bash
node migrate.js --dbuser=<user> --dbpass=<password> --dbname=<database>
```

Запустіть застосунок:
```bash
node mywebapp.js --port=8000 --dbuser=<user> --dbpass=<password> --dbname=<database>
```

### 2.4. Документація по API-ендпоінтах
* **GET /** — Кореневий ендпоінт. Віддає text/html зі списком доступних маршрутів.
* **GET /items** — Вивести список усіх предметів в інвентарі (id, name). Підтримує JSON та HTML.
* **POST /items** — Створити новий запис (очікує параметри name та quantity). Підтримує JSON та HTML.
* **GET /items/:id** — Вивести детальну інформацію по запису (id, name, quantity, created_at). Підтримує JSON та HTML.
* **GET /health/alive** — Перевірка стану (повертає 200 OK).
* **GET /health/ready** — Перевірка підключення до БД (повертає 200 OK або 500 Error).

## 3. Документація по розгортанню

### 3.1. Базовий образ віртуальної машини
* Для розгортання використовується офіційний образ Debian 12.
* Рекомендується завантажити debian-12.x.x-amd64-netinst.iso з офіційного архіву образів Debian (версія трошки застаріла) https://cdimage.debian.org/cdimage/archive/
* Перейдіть за посиланням, знизу у списку папок знайдіть одну з версій 12-ї гілки (наприклад, 12.8.0 або 12.9.0).
* Зайдіть в цю папку, потім натисніть amd64 -> iso-cd.
* Там, у самому низу списку файлів, буде лежати debian-12.x.x-amd64-netinst.iso

### 3.2. Вимоги до ресурсів віртуальної машини
* **CPU:** Мінімум 2 ядра.
* **RAM:** 2048 MB (рекомендується 4096 MB для стабільної роботи GUI).
* **Disk:** Мінімум 10 GB.
* **Video:** 128 MB.

### 3.3. Встановлення ОС
Спеціальних налаштувань не потрібно. Встановлення проводиться за замовчуванням (з підключенням до мережі Інтернет для завантаження пакетів). Під час встановлення створюється користувач vboxuser (або інший), який згодом буде заблокований автоматизацією.

### 3.4. Вхід на ВМ та credentials
Вхід здійснюється через консоль VirtualBox (GUI).
* **До виконання автоматизації:** користувач root (або ваш дефолтний користувач) із вказаними вами при встановленні паролями.
* **Після виконання автоматизації:** вхід здійснюється за допомогою створених користувачів (дефолтний користувач блокується). Пароль для всіх нових користувачів за замовчуванням — 12345678 (потребує зміни при першому вході).

### 3.5. Як запустити автоматизацію розгортання
Увійдіть в систему під користувачем root (або через sudo) та виконайте:

```bash
1. Встановіть git:
apt update
apt install -y git

2. Склонуйте репозиторій:
git clone https://github.com/yusik100/2-course-trpz.git

3. Перейдіть до папки скрипта і запустіть його:
cd 2-course-trpz/lab1
bash setup.sh
```

Скрипт автоматично встановить Nginx, MariaDB, Node.js, створить користувачів, налаштує Systemd Socket Activation та Reverse Proxy.

## 4. Інструкція з тестування розгорнутої системи

Після успішного виконання скрипта setup.sh, коректність налаштувань можна перевірити наступними кроками:

**1. Перевірка роботи застосунку та Nginx (Reverse Proxy):**
`curl -H "Accept: application/json" http://localhost/items`
Очікуваний результат: Повернення пустого масиву [] або списку об'єктів у форматі JSON. Це підтверджує, що Nginx (порт 80) успішно проксіює запити на Socket (порт 8000), який пробуджує застосунок і підключається до БД.

**2. Перевірка безпеки Nginx:**
`curl -I http://localhost/health/alive`
Очікуваний результат: Помилка 403 Forbidden (доступ до health-ендпоінтів ззовні заблоковано).

**3. Перевірка Systemd Socket Activation:**
`sudo systemctl status mywebapp.socket`
Очікуваний результат: Статус active (listening) і вказівка Triggers: mywebapp.service.

**4. Перевірка користувачів та їх прав:**
* **su - student:** Має перевести на користувача, вимагати зміну пароля. Доступна команда sudo ls /root. У домашній директорії знаходиться файл gradebook із цифрою 8.
* **su - teacher:** Вимагає зміну пароля. Має права адміністратора (sudo). Файл gradebook відсутній.
* **su - operator:** Дозволено виконувати команду sudo systemctl reload nginx та керувати станом сервісу mywebapp без введення пароля. Спроба виконати sudo ls /root або sudo systemctl stop nginx блокується системою (Permission denied).
* **sudo su - app:** Миттєве закриття сесії без помилок доступу, оскільки користувач має оболонку /bin/false (захист від входу людини), але від його імені успішно працює фоновий процес застосунку (ps -u app -f).
